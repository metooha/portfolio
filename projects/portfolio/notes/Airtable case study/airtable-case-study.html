01 · Situation

No shared source of truth. Everyone was working alone.

Design

Figma was restricted

The core team locked down their files — no direct access, no token contributions, no syncing. Figma couldn't be a source of truth.

Engineering

GitHub was fragmented

iOS, Android, and Web each maintained their own token repos with no shared inheritance model. Each platform was an island.

Commerce

Commerce needs were unmet

Commerce teams needed branding tokens, extended components, and multi-org support — none of which the core system was scoped to provide.

02 · The Human Cost

The cost wasn't just inefficiency. It was eroded trust.

  • Teams were doing the same work twice — every team kept their own copy of the same tokens.
  • Designers and engineers couldn't trust they were looking at the same thing. Decisions made in design were constantly re-litigated during implementation.
  • Commerce teams had no clear place to go for answers, so they built in isolation — creating more inconsistency and more overhead with every release.
  • Manual exports took 2–3 days per update with a ~15% error rate, and rollbacks were painful for everyone involved.
"The cost wasn't just inefficiency. It was eroded trust between teams."

03 · The Work

Three phases. Eighteen months. One source everyone could trust.

Phase 1
Q3–Q4 2024
Foundation

Establish alignment — not just tooling

The first challenge was getting people to agree before anything was built. Engineering wanted code-native workflows. Design wanted visibility and simplicity. The core team had restrictions. Platform teams had their own habits.

I mapped the full stakeholder landscape across decision makers, platform leads, subsystem designers, and engineering collaborators, then ran structured working sessions to surface the real blockers. Partnered with Shawn (WCP engineering lead) on a proof of concept that demonstrated tokens could be pulled from Airtable via API and consumed by platform tooling. This gave us something concrete to bring to leadership instead of an abstract debate.

Key outcomes: Airtable established as SoT, token taxonomy and IA defined across all platforms, RACI model locked in to close the ownership gap that had been causing friction for months.

Phase 2
Q1–Q2 2025
Automate

Remove the manual steps creating the most friction

Once Airtable was the SoT, the next problem surfaced: it was still too manual. Setting up a new tenant took 4 hours. Human errors were causing script failures. New team members couldn't onboard without extensive hand-holding.

Restructured the Airtable IA to remove legacy bloat and standardize naming conventions — base name confusion was the single biggest source of human error. Designed user flows for a designer-friendly form for tenant onboarding. Worked with Zach to define an intermediate JSON representation so column changes in Airtable wouldn't create breaking changes in engineering scripts.

Defined the theme inheritance model: each tenant contains only overrides, with a hierarchy file providing inheritance rules (Sam's Club ← WCP ← LD Base).

Phase 3
Q3–Q4 2025
Ship

Close the loop — end-to-end automation

Worked with Zach to implement a fully automated CI/CD pipeline: Airtable → Style Dictionary → GitHub Actions → Artifactory/npm. The pipeline handles token validation, semantic versioning, and multi-channel distribution. 347 automated tests run on every sync.

Asset tokens (icons, SVGs, logos) were also brought into the pipeline — a different class of problem requiring optimization (SVGO), platform-specific conversion (PDF for iOS, VectorDrawables for Android), and versioned packaging alongside JSON.

If a token breaks — an alias pointing to nothing, a hardcoded hex where a token reference should be — the pipeline stops and designers get a Slack alert within 5 minutes.

05 · Theme Hierarchy

17 brands. One base. Each layer only defines its overrides.

01
LD Base — Primitives
650+ tokens. Raw values — hex colors, px scale, font sizes. No semantic meaning. Shared by every brand that inherits from Living Design.
100% shared
02
Semantic Layer
Alias tokens that map intent to primitives — e.g. colorTextDefault → gray-900. Shared across all brands inheriting from LD Base.
~90% inherited
03
WCP Theme
Commerce-specific overrides only. Sam's Club, Walmart+, and Bodega all branch from this layer. Only what's different gets defined.
~20% override
04
Tenant Overrides
Brand-specific. Cashi, Walmart Business, international markets. Each tenant contains only its unique overrides — zero duplication.
~10% override

When LD Base updates, all 17 downstream tenants receive the change automatically on the next sync — no manual propagation required.
